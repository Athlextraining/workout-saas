# DEC Chain — Design Spec

Date: 2026-06-20
Status: Approved

## Purpose

The repo is opening to outside collaborators. We need a lightweight, git-native
way to record **decisions + the work they spawn**, with status tracking and
ownership. A "DEC" (Decision) is a single numbered record that captures *why* a
choice was made, *what* will be done, *who owns it*, and *where it stands*.

This spec defines the on-disk format and four user-invocable skills that manage
the DEC chain.

## File layout

- `DECISIONS.md` — root of repo. The single chain file: a status board table on
  top, followed by one detailed block per DEC, in numeric order.
- `contributors.md` — root of repo. Table of valid assignees.
- `.claude/skills/dec-creation/SKILL.md`
- `.claude/skills/dec-amend/SKILL.md`
- `.claude/skills/dec-deletion/SKILL.md`
- `.claude/skills/dec-close/SKILL.md`

Each skill has frontmatter `name`, `description`, `user_invocable: true`,
matching the existing `generate-template-cycle` skill convention.

## `DECISIONS.md` structure

### Status board (top of file, auto-maintained)

Regenerated by every skill that mutates the chain. Gives the whole "chain" at a
glance in one file.

```
| DEC     | Title                  | Status      | Assignee |
|---------|------------------------|-------------|----------|
| DEC-001 | Adopt DDD bounded ctx  | Done        | @patri   |
| DEC-002 | Switch to pnpm         | In Progress | @handle  |
```

Rows appear in numeric order and include every DEC regardless of status
(Rejected/Superseded rows stay listed, with their status shown).

### DEC block (one per DEC, below the board)

```
## DEC-001 — Short title

- Status: Pending
- Assignee: @handle
- Created: 2026-06-20
- Branch: dec-001-short-title
- Supersedes: —
- Superseded by: —
- Related: —

### Context

Why this decision is needed.

### Decision

What we will do.

### Tasks

- [ ] First task
- [ ] Second task
```

Field rules:
- **Status**: one of the six statuses below.
- **Assignee**: a single `@handle` present in `contributors.md`, or `Unassigned`.
- **Created**: ISO date `YYYY-MM-DD`.
- **Branch**: convention `dec-NNN-slug` (set at creation; used by `/dec-close`).
- **Supersedes / Superseded by**: bidirectional links to other DEC codes, or `—`.
- **Related**: optional comma-separated DEC codes, or `—`.
- **Tasks**: a GitHub-style checklist. A single DEC groups multiple related
  tasks under one number, one assignee, one status.

## Statuses

| Status      | Meaning                                                        |
|-------------|----------------------------------------------------------------|
| Pending     | Created, not started.                                          |
| In Progress | Work underway.                                                 |
| Blocked     | Waiting on another DEC or an external dependency.              |
| Done        | Merged to main and complete (set by `/dec-close`).            |
| Rejected    | Cancelled via `/dec-deletion`. Kept in place, never reused.   |
| Superseded  | Contradicted by a newer DEC. Links to the DEC that replaced it.|

## Numbering

- Codes are zero-padded: `DEC-001`, `DEC-002`, …
- Next number = `max(existing) + 1`, counting **all** DECs including Rejected and
  Superseded ones.
- Numbers are **never reused and never renumbered**. Deletion only changes status.

## `contributors.md`

```
# Contributors

| GitHub  | Name            | Role      |
|---------|-----------------|-----------|
| @patri  | Patricio Guerra | Maintainer|
```

`/dec-creation` and `/dec-amend` validate any assignee against this table.
`Unassigned` is always permitted.

## Skills

### `/dec-creation`

1. Read the user's prompt describing the decision/work.
2. Scan **active** DECs only (skip `Rejected` and `Superseded`) for a
   contradiction with the new proposal. This is a best-effort semantic check; the
   result is surfaced to the user, never auto-applied.
3. If a contradiction is found, ask the user:
   *"This contradicts DEC-xxx. Supersede it?"*
   - **No** → stop. Create nothing.
   - **Yes** →
     - Old DEC: set `Status: Superseded`, set `Superseded by: DEC-<new>`.
     - New DEC: set `Supersedes: DEC-<old>` (bidirectional link).
4. Assign the next number. Ask for the assignee and validate against
   `contributors.md` (`Unassigned` allowed). Set `Status: Pending`.
5. Collect `Context`, `Decision`, and the `Tasks` checklist.
6. Set `Branch: dec-NNN-slug`. Offer to create that git branch now so the
   `/dec-close` convention works later.
7. Append the DEC block and regenerate the status board.

### `/dec-amend`

1. Ask which DEC to change: accept a DEC code directly, or a free-text
   description to search the chain by title/context.
2. Let the user edit any field, the task checklist, the status, the assignee, or
   the body. Re-validate the assignee against `contributors.md` if changed.
3. Regenerate the status board.

### `/dec-deletion`

1. Ask which DEC (code or description search).
2. Set `Status: Rejected`. Do **not** renumber or reorder anything.
3. Regenerate the status board.

### `/dec-close`

1. Read the current git branch and parse the `dec-NNN` prefix.
   - If on a `dec-NNN-*` branch, use that DEC.
   - If already on `main`, derive the DEC from the most recent merge commit
     message/branch; if ambiguous, ask the user for the DEC code.
2. Verify the work merged to `main`.
3. Set `Status: Done`. Tick any remaining unchecked tasks, or warn the user if
   tasks are still open and confirm before closing.
4. Record the merge commit hash and date in the DEC block (appended note line).
5. Regenerate the status board.

## Out of scope (YAGNI)

- No per-DEC files or directories (single chain file by decision).
- No automated PR/CI integration beyond reading local git state.
- No multiple assignees per DEC.
- No automatic contradiction enforcement — human confirms every supersede.
